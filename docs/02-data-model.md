# 데이터 모델 설계 — SAP MM 구조 차용

> "왜 이 구조인가"를 설명하기 위한 문서. (교수님 요구: 아키텍처 복잡도를 정당화할 것)

## 1. 왜 SAP MM 구조를 차용했나

일반적인 재고관리 토이 프로젝트는 `상품(수량)` 테이블 하나에 숫자를 더하고 빼는 식으로 끝난다.
하지만 실무 ERP(SAP MM)는 **"재고 수량"을 직접 수정하지 않는다.** 대신 모든 입출고를
**자재문서(Material Document)** 로 기록하고, 재고는 그 이동의 *결과로* 집계된다.

이 구조를 따르면 얻는 것:

- **감사 추적(audit trail):** 언제·무엇이·왜(이동유형) 움직였는지 전부 남는다. 재고가 틀려도 역추적 가능.
- **수요예측의 재료:** 출고 이력(MSEG, 이동유형 201) 자체가 곧 수요 시계열이 된다 → 5단계 회귀분석 입력.
- **현업 설득력:** SAP를 쓰는 기업 담당자에게 "MARA/MARD/MKPF/MSEG를 차용했다"는 한마디로 구조가 전달된다.

## 2. 테이블 구성 (11개)

| 분류 | 테이블 | SAP 원본 | 역할 |
|------|--------|---------|------|
| 조직 | `plant` | T001W | 사업장/물류센터 |
| 조직 | `storage_location` | T001L | 저장위치(창고) |
| 마스터 | `material_group` | T023 | 자재 그룹(난방/냉방/음료…) |
| 마스터 | `material` | MARA+MAKT | 자재 마스터 |
| 재고 | `stock` | MARD | 저장위치별 재고 + 안전재고/ROP |
| 이동 | `movement_type` | BWART | 이동유형(101 입고/201 출고/561 기초) |
| 이동 | `material_doc_header` | MKPF | 자재문서 헤더(전기일 BUDAT) |
| 이동 | `material_doc_item` | MSEG | 자재문서 항목(수량 MENGE) |
| NFC | `nfc_tag` | — | 태그 UID ↔ 자재 매핑 |
| 외부 | `ext_weather` | — | 일별 날씨(기상청) |
| 외부 | `ext_holiday` | — | 공휴일 |

컬럼은 가독성을 위해 영문 의미명을 쓰되, 원본 SAP 필드명(MATNR, BUDAT 등)을 주석으로 병기했다.
순수 SAP 약어(MATNR, LABST…)는 협업·발표 시 가독성을 해쳐서다.

## 3. 핵심 흐름: 입출고 → 재고

```
NFC 스캔 → 자재문서 생성 (MKPF 헤더 1건 + MSEG 항목 N건)
         → 이동유형 direction(+1/-1) × 수량 만큼 stock.unrestricted_qty 갱신
```

이동이 진실의 원천이고 재고는 파생값이다. (실무 SAP와 동일)

## 4. 단일 기업용 결정

멀티테넌트(`tenant_id`) 컬럼을 일부러 넣지 않았다. 교수님 피드백("SaaS 트렌드는 지났다",
단일 기업 백오피스에 집중)에 따른 의도적 단순화다. 조직 구조는 plant/sloc 2단계로 충분하다.

## 5. 시드 데이터 설계 (db/seeds/seed_transactions.py)

회귀분석이 의미 있으려면 **기온과 상관관계가 있는** 출고 이력이 필요하다. 그래서 시드는
1년치 날씨(사인 곡선 계절성)와, 자재별로 기온계수를 다르게 준 출고 이력을 생성한다.

- 난방용품(히터/매트/핫팩): 기온↓ → 출고↑ (음의 상관)
- 냉방용품(선풍기): 기온↑ → 출고↑ (양의 상관)
- 생수/물티슈: 계절 영향 적음 (대조군)
- 주말·공휴일: 수요 +25%

→ 5단계에서 "기온 -1℃ → 난방 출고 +x%" 같은 해석 가능한 회귀계수가 나오도록 의도.

## 실행 방법

```bash
docker compose up -d db                       # 스키마는 migrations/ 자동 적용
python db/seeds/seed_transactions.py          # 시계열 거래/날씨 생성
```
