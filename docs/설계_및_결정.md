# StockCast 설계 및 기술적 의사결정

공개용 설계 문서. 아키텍처, 기술 선택 이유, 데이터 모델, 연동·배포 구조를 정리한다.

## 1. 시스템 목표와 도메인

**공공조달 납품업체**용 백오피스(내부 운영 시스템). 고객 대면 기능은 없고, 재고·조달·수요예측·경영분석 같은 운영/관리 기능만 다룬다.

취급 품목 30종은 조달청·나라장터에 실재하는 카테고리(제설·방재, 안전·보호구, 냉난방, 사무, 청소·위생, 방역·보건, 제초·조경, 전기·조명)로 구성해, 외부 공공데이터(나라장터 입찰=실수요, 조달청 단가=실가격, 기상청 날씨=계절 수요)가 도메인과 정합하도록 설계했다.

## 2. 아키텍처 — 운영계(OLTP) + 분석계(OLAP) 분리

- **Odoo 18(실제 ERP) = 운영계(system of record):** 실시간 실물 재고·입출고·재주문 규칙.
- **StockCast(FastAPI + PostgreSQL) = 분석계(데이터 웨어하우스):** 1년치 거래 이력 + 외부 공공데이터를 모아 수요예측·안전재고·경영분석.

두 시스템은 **XML-RPC로 양방향 연동**한다.
- 정방향: StockCast 재주문점·발주상한 → Odoo 재주문 규칙(stock.warehouse.orderpoint)
- 역방향: Odoo 실재고(qty_available) → StockCast 대시보드

**DB를 둘로 나눈 이유:** ① 외부 데이터(날씨·공휴일·입찰·단가)는 Odoo에 저장할 자리가 없고, 예측은 "출고량 × 그날 날씨"를 조인해야 한다. ② 무거운 분석 쿼리를 실시간 운영 DB에 돌리면 입출고 업무 성능이 저하된다. ③ 매 조회마다 Odoo에서 1년치를 끌어와 회귀를 돌리면 느리다. → 현직 표준인 OLTP/OLAP 분리를 채택.

## 3. 기술 선택과 이유

| 기술 | 선택 이유 |
|------|----------|
| FastAPI | 자동 OpenAPI 문서, pydantic 타입검증, 비동기 성능 |
| SQLAlchemy 2.0 | ORM으로 DB 독립(운영 PostgreSQL / 테스트 SQLite), merge로 멱등 upsert |
| PostgreSQL 16 | 표준 RDB. 테스트는 SQLite 인메모리(BigInteger.with_variant로 PK 호환) |
| Odoo 18 Community | 무료 오픈소스 실제 ERP. 자체 호스팅이라 외부 API 제약 없음(온라인 무료판은 API 차단) |
| statsmodels | OLS 계수·p-value·R² 해석으로 "기온 1℃당 출고 X개" 의사결정 근거 제공 |
| React+Chart.js 단일 HTML | 빌드 도구 없이 백엔드가 서빙 → 인프라 단순화 |
| Docker/Terraform/Caddy | 환경 일관성·IaC·자동 HTTPS |

차별점: Odoo는 AI·수요예측이 Enterprise(유료) 전용이라, 그 기능을 StockCast가 무료로 대체한다.

## 4. 데이터 모델 (15개 엔터티, SAP MM 구조 차용)

- **마스터:** plant, storage_location, material_group(코드성), material, movement_type(코드성)
- **재고/문서:** stock, material_doc_header, material_doc_item
- **NFC:** nfc_tag(UID→자재 매핑)
- **이력성:** stock_snapshot_history(월말 재고 스냅샷 — 평균재고·회전율 근거)
- **외부 공공데이터:** ext_weather, ext_holiday(코드성), ext_bid_notice, ext_shop_price, ext_retail_index

입출고는 자재문서(헤더+품목) 전기 → 이동방향(±1)×수량으로 stock 갱신. 재고부족·미등록태그는 예외처리.

## 5. 실 공공데이터 커넥터 (공공데이터포털)

설계 공통: HTTP 호출부/파싱부 분리(파싱 단위테스트 가능), merge 멱등 upsert, 키 없으면 건너뛰고 안내(비용0·항상 동작).

1. 기상청 ASOS 일자료 — 일 기온·강수
2. 한국천문연구원 특일정보 — 공휴일(같은 날짜 중복은 dedup)
3. 나라장터 입찰공고(물품) — 실수요 신호(조회기간 30일)
4. 조달청 종합쇼핑몰 MAS — 실 계약단가

`scripts/collect_real_data.py`가 날씨·공휴일을 먼저 적재한 뒤 거래를 실제 날씨·공휴일에 반응하도록 생성한다. 커넥터가 실패해도 rollback 후 계속해 핵심 데이터는 항상 완성된다.

## 6. 분석 엔진

- **수요예측:** 다중회귀(OLS, 해석 가능) + 시계열(Holt-Winters/SARIMA). 분산0 변수로 통계가 NaN/Inf가 되면 None으로 안전 처리.
- **재고:** 안전재고 SS=Z·σ·√L, 재주문점 ROP=평균일수요·L+SS, 발주상한=평균일수요·(L+검토주기)+SS.
- **경영지표:** ABC(매출 파레토), 재고회전율, 결품률, 재고자산금액(운전자본).

## 7. ERP(Odoo) 연동 — XML-RPC

- 인증(common)→uid, 실행(object/execute_kw).
- 백엔드 컨테이너→Odoo는 host.docker.internal:8069(리눅스는 compose extra_hosts host-gateway).
- 정방향: 재주문 규칙 write-back. 역방향: 실재고 read. NFC: 태그→품목→stock.quant 입출고.
- 주의점: 검색 도메인 단일 래핑, None 반환 메서드(action_apply_inventory)는 직렬화 불가라 quant 수량 직접 설정, Odoo18 저장가능 품목은 type=consu+is_storable.

## 8. NFC

각 태그 UID를 품목에 매핑. 실물 폰 태깅(Web NFC, 안드로이드 크롬, HTTPS 필요) 또는 등록 태그 클릭(데모). 스캔 → Odoo 실재고 입/출고. HTTPS는 Caddy로 확보(Web NFC는 보안 컨텍스트 필수).

## 9. 화면

React 단일 HTML. 헤더 매니저 바(KPI / Odoo 실재고 / NFC 입출고 + Odoo ERP 바로 열기). KPI 탭은 AI요약·KPI 카드·차트·ABC·발주권고. NFC 입출고는 항상 운영계(Odoo)로 반영.

## 10. 인프라·배포

- Docker Compose(StockCast / Odoo / Caddy 분리 스택).
- Terraform: VPC·보안그룹(22/8000/8069/80/443)·EC2(t3.small)·Elastic IP(중지/시작해도 주소 유지).
- Caddy + Let's Encrypt 자동 HTTPS, 무료 DuckDNS 도메인.
- 비용 최소화: 안 쓸 때 인스턴스 중지하면 컴퓨팅 0, 고정IP로 주소 유지.

## 11. 품질

pytest 54건(SQLite 인메모리). 통합테스트로 시드 FK 위반·예측 NaN 직렬화·수집 견고성·공휴일 PK 충돌 등을 해결.

## 12. 한계와 개선 방향

수요는 모델 생성치(실 판매 트랜잭션 부재), 인증/권한·감사로그 미구현(PoC 범위), 단일 노드. 개선: 실거래 연동, RDS 분리·다중 AZ, 예측 모델 고도화, 시크릿 매니저, RBAC.
